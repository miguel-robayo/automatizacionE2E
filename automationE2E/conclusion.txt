      - No se detectan validaciones fuertes para completar la compra de los productos asi como los tipos de datos que

CONCLUSIONS

  The end-to-end automation flow for purchasing technology products was successfully implemented, ensuring happy path
   coverage and regression risks. The automation addressed:

    - Selection of multiple products
    - Shopping cart management
    - Checkout process
    - User information input
    - Confirmation message validation

  At a technical level, the following points were identified:

    - Xpath locators were used to identify web elements. It is recommended to prioritize By.id() whenever possible in order
    to improve stability and performance.
    - Strong validation were not detected for completing the purchase process, particularly regarding required fields and
    data types allowed in each input field.

The framework architecture was strengthened through:

    - Implementation of the Screemplay pattern
    - Clear separation of responsibilities (Page, Task, Question, Step Definition)
    - Data parameterization from the .feature file (List, DataTable)
    - Improvements in maintainability, readability, and reduction of code duplication

Improvement Opportunities

    - Validation of total amounts and product quantities
    - Negative validations (empty fields, invalid data)
    - Error handling scenarios
    - Validation of the generated purchase ID after completing the transaction
    - CI/CD pipeline integration
    - Execution by tags (regression, smoke)

Issues Encountered

    - Outdated Java and Maven versions
    - Dependency management challenges across different projects (E2E and API)
    - Browser alerts (JavaScript) not handled by the automation
    - SSH key generation required to push code to the GitHub repository
